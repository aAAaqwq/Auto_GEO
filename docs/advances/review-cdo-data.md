# CDO 数据架构审查报告

_审查日期: 2026-05-06 | 审查人: CDO_

---

## 综合评分：6.5/10

## 🔴 核心问题

### 1. SQLite 生产环境风险
- 开发期可行，生产环境约 75% 概率出现锁竞争
- WAL 模式缓解但不能根治并发写入瓶颈
- **建议**: 评估 PostgreSQL 迁移，至少做压力测试验证

### 2. GeoArticle 表职责过重
- 单表承担过多业务逻辑（文章状态、发布状态、SEO 数据等）
- 字段膨胀，查询效率下降
- **建议**: 拆分为 article + article_publish_status + article_seo_metrics

## 🟡 数据质量问题

### 3. status 字段类型不统一
- 不同表 status 用不同类型（int/string/enum）
- 维护和查询复杂度高
- **建议**: 统一使用 enum 或 string 常量

### 4. 废弃字段未清理
- 存在不再使用的字段，增加理解和维护成本
- **建议**: 通过迁移清理废弃字段

### 5. 双路径关联
- 部分数据存在两种关联方式，容易不一致
- **建议**: 确定唯一关联路径，清理冗余关联

## 🟡 迁移管理

### 6. 仅 1 个初始迁移
- 无增量迁移纪律，schema 变更无追踪
- **建议**: 建立迁移规范，每次变更生成迁移文件

## 🟡 RAGFlow 集成

### 7. 两套重复客户端
- `ragflow_client.py` 和 `ragflow_integration_colleague.py` 功能重叠
- 文档更新策略有缺陷
- **建议**: 合并为统一客户端，明确文档同步策略

## ✅ 做得好的
- 15 张表覆盖完整 GEO 链路
- 数据模型与业务流程对齐
- RAGFlow 集成方向正确（作为知识库主存储）
